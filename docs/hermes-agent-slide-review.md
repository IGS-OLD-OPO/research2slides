# Hermes Agent 投影片審查紀錄

> 審查日期：2026-06-23
> 審查對象：`decks/hermes-agent/slides.md`（29 張）
> 研究目的：評估 Hermes Agent 適合拿來做什麼、企業導入是否有資安風險

## 整體評估

架構上「技術介紹 → 資安 → 案例 → 結論」是合理的。實際案例有真實數據，是最強的部分。
但作為評估報告，目前更像「Hermes Agent 功能介紹」，而非提供明確判斷的「技術評估」。

---

## 邏輯安排問題

### 1. 開場過長（P2–P3，神話暖場）

兩張投影片在講 Hermes 的名字典故，佔有限注意力預算。作為評估報告對決策者沒有幫助判斷的作用。
**建議**：移除 P2（純 teaser），保留 P3 即可。

### 2. 「六大核心技術特色」(P10) 是重複的

P11–P19 每個特色都有獨立投影片詳細展開。P10 等於先列了一份等一下要重講的目錄，沒有新增資訊。
**建議**：移除，直接進入第一個特色。（尚未執行，可視情況決定）

### 3. 案例細節太深（P24–P26）

DailyUserInfoSnapshot 欄位清單、SQL 範例、業務縮寫字典——這三張是在說「我們怎麼設定知識庫」，
不是在說「Hermes Agent 能做什麼」。另外也暴露了內部資料結構。
**處置**：主管表示想看 example，移至 Appendix 保留，不從簡報主體刪除。

### 4. 結論（P27）太薄

結論只有通用優劣條列 + 一行建議，沒有直接回答研究目的的兩個問題：
- Hermes Agent 適合拿來做哪些事？
- 企業導入的資安風險如何、如何管理？
**建議**：拆成「能做什麼 / 資安結論 / 下一步」三欄，每欄給具體答案。

### 5. 資安的位置略奇怪

P14（Promptware）→ P15（資安全貌）→ P16（Multi-agent Kanban）——資安結束後又回去講技術特色，
讀起來像技術特色章節切了一半又回來接。
**建議**：P15 資安全貌移到技術特色章節末尾（P19 企業可靠性之後），讓資安做為所有技術特色的總結。

## 已補上的部分

### 指令審批機制（Command Approval）

Hermes 有 4 種審批模式（auto / suggest / ask / deny）+ 白名單/黑名單，
控制 Agent 能執行哪些指令，是企業資安控管的核心機制。
**位置**：插在 P19 企業可靠性之後、P15 資安全貌之前。

### 適用場景 vs 不適用場景

明確說明 Hermes 適合與不適合的企業場景，直接回答評估目的。
**位置**：插在 P15 資安全貌之後，作為技術評估章節的收尾。

## 目前著重點與初衷的落差

| 面向 | 目前張數 | 評估 |
|------|----------|------|
| 技術特色介紹 | ~9 張 | 偏多，著重描述而非判斷 |
| 案例細節 | 7 張 | 太多，實作細節移 Appendix |
| 資安 | 2 張 | 深度不錯，位置已調整 |
| 結論 | 1 張 | 太薄，已加強 |

**核心落差**：缺乏「判斷」視角——評估報告要有是否、適合/不適合的明確答案，不只是功能描述。
資安兩張（P14、P15）是最像評估報告的部分；案例也很有力，但細節要收到 Appendix。
